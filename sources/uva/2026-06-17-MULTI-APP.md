# Multi-Project Living Wiki — app overview

This repo is the **multi-project** version of the Living Project Wiki app. It runs the same
four operations (ingest · query · generate · lint) and the same permission/gap-detector
machinery as the original single-project app, but hosts **several projects side-by-side in
one app**, each with its own isolated knowledge base, selected by an in-app toggle.

See the decision record: [[decision-multi-project-app-structure]] (in `wiki/uva/decisions/`).

---

## Why it exists

The original app assumed **one wiki = one project** (a flat `/wiki` + `/sources`). Two needs
pushed beyond that:

1. **Reusability across KickstartAI projects** — the artifact is meant to be deployed to many
   projects (UvA, Bakkie, …). Rather than fork the app per project, one app serves all of
   them, each isolated, switchable from a toggle.
2. **A clean-slate corpus for the thesis evaluation** — a brand-new, empty project subtree
   can be ingested from scratch in a controlled way (known provenance + per-ingest token
   numbers), which the long-lived original wiki can't give.

---

## What's different from the single-project app

| | Single-project (original) | Multi-project (this app) |
|---|---|---|
| Layout | flat `/wiki`, `/sources` | per-project subtrees `wiki/<project>/`, `sources/<project>/` |
| Operation scope | implicit (the one project) | every call carries a `project` (`?project=<id>`) |
| Config | hard-coded | `backend/project_config.py` + optional `project.config.json` |
| Accounts | one project's users | users carry a `projects` list; access is per-project |
| Frontend | single view | **active-project switcher**; all views act on the selected project |
| Isolation | n/a | separate `index`/`_overview`/`_gaps`, sources, and access labels per project |

## Architecture

**Per-project subtrees.** Each project is a self-contained wiki + source tree:
```
wiki/uva/…       sources/uva/…       ← Living Wiki (UvA)
wiki/bakkie/…    sources/bakkie/…    ← Project Bakkie
```
`backend/main.py::project_dirs(project)` resolves `(wiki_dir, sources_dir)` for a project and
validates the id against `permission.PROJECTS`. So each project has its own
`index.md`/`_overview.md`/`_gaps.md` and never bleeds into another.

**Config in one place.** `backend/project_config.py` is the single source of per-project
config — projects, demo accounts, and the git push target — read from `project.config.json`
at the project root, falling back to built-in `DEFAULTS` so the app always boots. A
`PROJECT_ROOT` env var can point the app at a different working tree. *Deploying to a new
project needs no code edits* — just config.

**Scoping the operations.** Every backend endpoint takes a `project` and routes through
`project_dirs` — `read_all_wiki(project)`, `retrieve_wiki_context(project, …)`,
`apply_ingest_files(files, project)`, etc. The frontend appends it via
`api.ts::withProject(path, project)` → `?project=<id>`.

**Permission & accounts.** Dummy accounts (in `project_config.py` / `permission.py`) carry a
`projects` list and a `tier`:
- `anna.jansen` → UvA · `bram.bakker` → Bakkie · `carla.visser` → both · `gast.bezoeker` → public guest (no projects).
Ingest/query/etc. are refused for a project the user isn't a member of.

**Frontend.** `App.tsx` keeps an **active project** in state (persisted to `localStorage`),
shown as a switcher (bottom-left, so it doesn't cover the nav). Changing it re-scopes every
view (Wiki, Operations, Gaps, Artifact) and drops stale per-project deep-links. `BACKEND_URL`
is configurable via `VITE_BACKEND_URL` (default `http://localhost:8000`).

## How it was built (in order)

1. **Initial scaffold** — multi-project skeleton (UvA + Bakkie), empty per-project trees.
2. **Project-scoped backend** — ingest + view per subtree, then query/lint/generate/gaps.
3. **Frontend active-project context** — switcher + `withProject` across all views.
4. **Clean slate** — reset both projects to empty skeletons for a controlled ingestion test.
5. **Configurable backend URL** (`VITE_BACKEND_URL`) and switcher moved to bottom-left.
6. **Ingestion cost instrumentation** — per-ingest token logging + prompt caching
   (`token_usage.md`), ported from the main repo.

## Running it

```bash
# backend (port 8000)
cd backend && python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload --port 8000     # run from backend/ (flat imports)

# frontend (port 5173)
cd frontend && npm install && npm run dev
```
`ANTHROPIC_API_KEY` is read from the root `.env`. Log in with a demo account (e.g.
`carla.visser@kickstartai.demo` / `wiki-demo-3` — sees both projects), pick a project from
the switcher, and ingest/query/generate against that project's subtree.

## Adding a new project

1. `mkdir -p wiki/<id> sources/<id>` (with the empty wiki skeleton files).
2. Add the project + any accounts to `project.config.json` (or `project_config.py` DEFAULTS)
   and to `permission.PROJECTS`.
3. No other code changes — the operations are already project-generic.

## Relation to the thesis / evaluation

The UvA project here is the **clean-slate corpus** for the wiki-vs-raw-context experiment:
a fresh `.md` ingest builds `wiki/uva/` from `sources/uva/`, the new token logging records the
**build cost** in `token_usage.md`, and the evaluation runs with `--project uva`. See
[`evaluation/RUN-PLAN.md`](evaluation/RUN-PLAN.md).
